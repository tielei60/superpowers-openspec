# 中文 OpenSpec 模板

> 用于说明规范文档应该写成什么样。

## 适用范围

适用于需要先分析、先写 spec、先做详细设计再开发的任务，也适用于把会议纪要、口语需求或草稿整理成可执行规范的场景。

## 阶段边界

本 skill 只负责：

1. 分析
2. 规范
3. Spec Review

后续拆解、实现、测试、最终 review 由顶层 `superpowers` 继续推进。

## Spec First

只要任务涉及功能、流程、页面、接口、数据结构、状态流转、业务规则或模块边界，就先写 spec，再写代码。

如果用户催促直接实现，也要先补最小可行 spec。
如果用户只要求规范产物，spec 完成后就停在规范阶段，不要继续拆任务或写实现建议。

## OpenSpec 结构

默认按以下顺序写作：

1. 文档头信息
2. 背景与目标
3. 范围与非范围
4. 角色与对象定义
5. 核心规则
6. OpenSpec 需求条目
7. 数据与结构定义
8. 接口与交互约束
9. 架构图（Mermaid）
10. 流程图（Mermaid）
11. 时序图（Mermaid）
12. 异常与边界处理
13. 验收标准
14. 待确认项
15. Spec Review 记录
16. 文档自检结果

如用户明确要求缩减，可以压缩篇幅，但不要省略对开发必要的关键信息。

## 写作要求

- 全文使用中文，技术名词和字段名可保留原文。
- 需求表达要具体，避免模糊词。
- 规则要写清触发条件、执行动作、输出结果和判断标准。
- 遇到现有系统、接口、状态或页面冲突时，先写差异，再写新规则。
- 涉及前端页面时，可补 ASCII 布局图，但不强制。

## 图表要求

规范文档应包含三类 Mermaid 图：

- 架构图：说明参与方、模块边界和数据流向。
- 流程图：说明主流程、分支、成功路径和失败路径。
- 时序图：说明调用顺序、关键交互和异常返回。
