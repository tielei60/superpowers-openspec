# OpenSpec 使用命令示例

> 用于告诉调用者应当怎样明确地使用 `superpowers-openspec`。

## 一、最常用命令

以下命令适合直接复制给上层 agent 或用户：

### 1. 产出 spec 初稿

`请使用 superpowers-openspec，基于下面的需求产出中文 OpenSpec 初稿。`

适用场景：

- 已经有较完整的需求描述
- 需要快速进入规范阶段
- 希望先看到第一版结构化产物

### 2. 先补信息再写 spec

`请使用 superpowers-openspec，先列出已知事实、缺失信息和待确认项，再输出中文 OpenSpec。`

适用场景：

- 需求来源零散
- 输入是聊天记录、口头描述、会议纪要
- 需求中存在明显空白

### 3. 把会议纪要整理成规范

`请使用 superpowers-openspec，把下面的会议纪要整理成可执行的中文 OpenSpec，并补齐关键规则、异常处理和验收标准。`

适用场景：

- 会议纪要缺乏结构
- 决策分散在多个段落里
- 需要把讨论结果沉淀为开发输入

### 4. 做变更 spec

`请使用 superpowers-openspec，先分析现有流程/接口/状态与新需求的差异，再输出变更版 OpenSpec。`

适用场景：

- 不是从零开始，而是改现有系统
- 需要重点说明兼容性和冲突
- 需要让开发清楚哪些地方被影响

### 5. 只做 Spec Review

`请使用 superpowers-openspec，对下面这份 spec 做 review，检查结构完整性、模糊描述、异常处理、Mermaid 图和验收标准，并给出明确结论。`

适用场景：

- 已经有初版 spec
- 需要独立的 review 结论
- 需要判断是否可以进入下一阶段

## 二、推荐补充参数

为了让输出更稳定，建议在命令里补充以下限制：

- `全文使用中文`
- `保持 OpenSpec 结构完整`
- `补齐架构图、流程图、时序图`
- `列出范围与非范围`
- `明确异常与边界处理`
- `附上验收标准`
- `最后给出 Spec Review 结论`
- `spec 完成前不要进入实现建议`

示例：

`请使用 superpowers-openspec，先列出已知事实、缺失信息、待确认项，再输出全文中文的 OpenSpec；需要包含范围与非范围、核心规则、Mermaid 三图、异常处理、验收标准，并在最后给出 Spec Review 结论；在 spec 完成前不要进入实现建议。`

## 三、输入类型与推荐命令

### 输入是口语需求

推荐命令：

`请使用 superpowers-openspec，把下面的口语需求整理成中文 OpenSpec；先补齐事实和待确认项，再形成规范。`

### 输入是会议纪要

推荐命令：

`请使用 superpowers-openspec，把会议纪要转成可执行规范，并显式标出已决策项、待确认项和验收标准。`

### 输入是旧系统改造需求

推荐命令：

`请使用 superpowers-openspec，先写旧规则与新规则的差异，再输出变更 spec，并说明受影响的接口、页面、状态和流程。`

### 输入是已有 spec 初稿

推荐命令：

`请使用 superpowers-openspec，对这份 spec 补齐缺失章节和 Mermaid 图，然后执行一次 Spec Review。`

## 四、不推荐的命令方式

以下表达容易让流程跑偏，应避免：

- `直接写代码，顺便补个 spec`
- `先给我实现方案，后面再说规范`
- `简单整理一下，不需要边界和异常`
- `先拆任务再补详细设计`

原因：这些说法会弱化规范阶段门禁，不符合 `superpowers-openspec` 的目标。

## 五、完成标准

当使用命令触发本 skill 后，理想输出通常应包含：

1. 已知事实
2. 缺失信息 / 待确认项
3. OpenSpec 正文
4. Mermaid 三图
5. 异常与边界处理
6. 验收标准
7. Spec Review 结论

若只完成其中一部分，应该明确说明当前仍处于规范阶段，尚未进入实现。
